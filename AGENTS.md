# AGENTS.md — Coding Agent Guidelines for Synthony

This file is the canonical reference for AI coding agents (Claude Code, Cursor, Cline, Continue.dev, Copilot, etc.) working in this repository. It complements `CLAUDE.md` (human-readable architecture reference) by focusing on operational rules, safe mutation patterns, and common workflows.

---

## First: Read Before Acting

Always read these files before making any change:

1. `CLAUDE.md` — full architecture, data flow, thresholds, key file index
2. `config/model_capabilities.json` — current registry version and all model scores
3. `config/SystemPrompt.md` — current LLM prompt version
4. `docs/scoring_methodology.md` — scoring formula and empirical validation rationale

## Repository Layout (Quick Reference)

```
src/synthony/
  core/analyzer.py          StochasticDataAnalyzer — entry point for profiling

mcp_server/
  server.py                 SynthonyMCPServer class (stdio JSON-RPC 2.0)
  tools/                    One module per tool group
```

## Invariants — Never Violate

1. **Both JSON files must stay identical.**
   `config/model_capabilities.json` and `src/synthony/recommender/model_capabilities.json` are always written together. If you edit one, edit the other. Use `synthony_update_model_capabilities` (MCP tool) for programmatic updates — it writes both atomically.

2. **MCP tool names use the `synthony_` prefix.**
   All tools exposed by the MCP server are named `synthony_<action>`. Do not add tools without this prefix.

3. **ToolAnnotations are required on every MCP tool.**
   Every `Tool(...)` definition must include an `annotations=ToolAnnotations(...)` block. Set `readOnlyHint`, `destructiveHint`, `idempotentHint`, `openWorldHint` accurately.

4. **SystemPrompt is generated, not hand-edited.**
   Dynamic sections (1, 2, 3, 7) of `config/SystemPrompt.md` are regenerated from `model_capabilities.json` by `synthony_update_system_prompt`. If you update capability scores, run the prompt regeneration tool afterward. Never manually patch the table rows inside the prompt.

5. **`EngineConfig` is loaded from the registry, not hard-coded.**
   Thresholds in `engine.py` (`EngineConfig`) read `metadata.capability_thresholds`, `metadata.dp_threshold`, `metadata.hard_problem_confidence`, and `metadata.score_decay` from the JSON registry. Change thresholds there, not in Python.

6. **The `DatasetProfile` schema is Pydantic v2.**
   All additions to `core/schemas.py` must use `pydantic.BaseModel` with proper type annotations. No `__fields__` usage.

## Capability Score Update Workflow

Use the MCP tools when possible; this is the safe path:

```
1. synthony_update_model_capabilities(model_name, spark_empirical, auto_calculate=true)
   → writes both JSON copies, bumps patch version, returns diff

2. synthony_update_system_prompt(version="vX.Y", set_active=true, version_note="...")
   → regenerates dynamic sections of SystemPrompt.md, stores in DB

3. Run: pytest tests/regression/ tests/unit/ -v
   → verify no scoring regressions
```

If editing JSON directly, keep `metadata.version` bumped (patch for score tweaks, minor for schema additions) and `metadata.last_updated` set to today.

## Adding a New MCP Tool

```python
# 1. In mcp_server/tools/<group>_tools.py

def get_tool_names(self) -> List[str]:
    return [..., "synthony_your_new_tool"]

def get_tool_definitions(self) -> List[Tool]:
    return [...,
        Tool(
            name="synthony_your_new_tool",
            description="...",
            inputSchema={"type": "object", "properties": {...}, "required": [...]},
            annotations=ToolAnnotations(
                readOnlyHint=True,
                destructiveHint=False,
                idempotentHint=True,
                openWorldHint=False,
            )
        )
    ]

async def execute_tool(self, name, arguments):
    ...
    elif name == "synthony_your_new_tool":
        return await self._your_new_tool(arguments)

# 2. Register in mcp_server/server.py under self.<group>_tools
# 3. Update CLAUDE.md MCP tool table
# 4. Update mcp_server/README.md tool documentation
# 5. Add a test in tests/unit/ or tests/integration/
```

## Adding a New Detector

Each stress detector lives in `src/synthony/detectors/`. The pattern is:

```python
# src/synthony/detectors/my_factor.py
class MyFactorDetector:
    def __init__(self, threshold: float):
        self.threshold = threshold

    def detect(self, df: pd.DataFrame) -> MyFactorResult:
        ...
        return MyFactorResult(detected=bool, ...)
```

Then:
1. Import and instantiate in `StochasticDataAnalyzer.__init__`
2. Call in `StochasticDataAnalyzer.analyze()` and merge result into `DatasetProfile`
3. Add field to `StressFactors` in `core/schemas.py`
4. Add unit tests in `tests/unit/test_my_factor_detector.py`

## Recommendation Engine: What Not to Touch

`engine.py` has a deliberate scoring pipeline order that must be preserved:

```
1. Load registry → 2. Hard filters → 3. Hard Problem check → 4. Capability score
→ 5. Empirical bonus → 6. Scale factors (focus) → 7. Tie-breaking → 8. LLM (optional)
```

If you change scoring logic, run the full evaluation suite afterward:
```bash
pytest tests/evaluation/ tests/regression/ -v
```
The evaluation tests measure top-3 accuracy against ground-truth spark benchmark rankings. A regression in `test_baseline_recommendations.py` means the scoring change broke known-good recommendations.

## Test Structure

| Path | What it covers |
|---|---|
| `tests/unit/` | Individual detectors and engine edge cases — fast, no I/O |
| `tests/integration/` | Full pipeline (CSV → profile → recommendation) — requires no external services |
| `tests/functional/` | CLI commands, REST API (requires `synthony-api` running), OpenAI (requires key) |
| `tests/evaluation/` | Recommendation accuracy vs trial4 ground truth |
| `tests/regression/` | Baseline recommendations and schema consistency |

Run the safe subset (no external services):
```bash
pytest tests/unit/ tests/integration/ tests/regression/ tests/evaluation/ -v
```

Skip tests that require live infrastructure:
```bash
pytest -m "not requires_llm" tests/
```

## Common Pitfalls

| Mistake | Correct Approach |
|---|---|
| Editing only one JSON file | Always edit both `config/` and `src/synthony/recommender/` copies |
| Hard-coding a threshold in engine.py | Add it to `model_capabilities.json` metadata and load via `EngineConfig` |
| Adding an MCP tool without `ToolAnnotations` | Always include `annotations=ToolAnnotations(...)` |
| Manually editing SystemPrompt table rows | Use `synthony_update_system_prompt` after updating the JSON |
| Changing `DatasetProfile` fields without migration | Update Pydantic schema, then check all serialization/deserialization paths |
| Running `pytest` and seeing API failures | Expected — `test_api_simple.py` requires a live server; skip with `-m "not requires_api"` |
| Committing with `Co-Authored-By: Claude Sonnet` | Do not include AI attribution in commit messages |

## Code Style

- **Formatter**: `black --line-length 100`
- **Linter**: `ruff` (E, F, W, I, N, UP rules)
- **Types**: `mypy --strict` on `src/`
- **Comments**: Only when the WHY is non-obvious. No docstrings that describe what the code already says.
- **Imports**: Absolute imports within the package (`from synthony.core.schemas import ...`)
